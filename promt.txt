━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

El bot está desplegado en Railway donde el sistema de
archivos es efímero. No se puede escribir en .env en
tiempo de ejecución.

Arquitectura de credenciales definitiva:
- .env y Railway → SOLO: SUPABASE_URL y SUPABASE_KEY
- Todas las API keys de Cerebras, Gemini y Groq →
  tabla api_keys en Supabase
- El bot las carga al arrancar desde Supabase
- Si no hay keys en Supabase → arrancar igualmente en
  PAUSADO con log claro: "No hay API keys. Añádelas
  desde Ajustes antes de activar el bot."

Contexto de modelos y límites:
- Gemini 2.0 Flash fue deprecado en febrero 2026 →
  migrar a gemini-2.5-pro
- Los límites de Gemini son por proyecto GCP, no por key
  → cada key debe venir de un proyecto GCP distinto
- Los límites de Groq son por API key → rotar keys
  multiplica la quota directamente
- Los límites de Cerebras son por organización (cuenta
  personal) → cada cuenta es una organización separada
- El usuario tendrá ~3 keys de cada proveedor

Nueva arquitectura de 3 modelos en cascada:
1. Cerebras (Qwen 3 235B) → screener primario ultra-rápido
2. Gemini 2.5 Pro → análisis profundo solo si Cerebras
   detecta gap >= 10%
3. Groq llama-3.3-70b → confirmación final solo si
   Gemini confirma gap >= 12% y confianza no es low
Un trade solo se abre si los 3 modelos coinciden.
Si los 3 discrepan más de 20% entre sí → no operar.

Sistema event-driven para el ciclo:
- Escaneo de precios: cada 5 minutos (sin LLM, barato)
- Re-análisis LLM: solo cuando ocurre alguno de estos:
  * Precio del mercado se movió > 5% desde último análisis
  * Apareció noticia nueva relevante en RSS
  * El mercado es nuevo (nunca analizado)
  * La caché LLM expiró (8 horas)
- Esto permite detectar movimientos rápido sin agotar
  las quotas de los proveedores LLM

Mercados: ampliar de 50 a 150 por ciclo de escaneo.
Caché LLM: 8 horas para los 3 modelos.
max_tokens: 400 en los 3 modelos.

PASO 1 — Crear core/key_manager.py
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Nuevo archivo que gestiona toda la lógica de rotación
de keys. El resto del código solo interactúa con este.

Funcionalidad:
- Al arrancar: cargar todas las keys habilitadas de
  Supabase agrupadas por servicio (cerebras/gemini/groq)
- get_next_key(service) → devuelve la key con menos
  calls_today, saltando las que están en cooldown
- mark_cooldown(key_id) → in_cooldown=True,
  cooldown_until = ahora + 30 minutos, loggear qué key
- mark_success(key_id) → actualizar last_used_at,
  incrementar calls_today y tokens_today en Supabase
- check_cooldowns() → liberar keys cuyo cooldown_until
  ya pasó, llamar periódicamente en el loop
- reset_daily_counts() → resetear calls_today y
  tokens_today a 0, llamar a medianoche Pacific Time
- reload_keys() → recargar desde Supabase sin reiniciar,
  llamar cuando el usuario añade/elimina keys
- Si todas las keys de un servicio están en cooldown →
  loggear warning y devolver None
- Las keys se guardan en memoria para no consultar
  Supabase en cada llamada LLM

full_analysis(market, market_price, news_articles,
              force=False):
  """
  Pipeline de 3 modelos en cascada.
  force=True ignora caché (para mercados con evento nuevo).

  1. Cerebras → screener primario
     - Si caché válida → usar caché
     - Si result es None o gap < 10% → return sin Gemini

  2. Gemini → análisis profundo
     - Solo si Cerebras detectó gap >= 10% y conf != low
     - Si caché válida → usar caché
     - Si result es None → continuar solo con Cerebras

  3. Groq → confirmación final
     - Solo si Gemini confirmó gap >= 12% y conf != low
     - Si caché válida → usar caché
     - Si result es None → continuar sin Groq

  Lógica de decisión:
  - gap_final = promedio de los modelos que respondieron
  - should_trade = True solo si:
    * Al menos 2 modelos respondieron
    * gap_final >= 15%
    * Ningún par de modelos discrepa > 20%
    * Ninguno devolvió confidence = low

  Retorna: (cerebras_result, gemini_result, groq_result,
            gap_final, should_trade)
  """

Sistema event-driven con dos loops separados en threads:

LOOP 1 — Escaneo de precios (cada 5 minutos, sin LLM):
- Obtener los 150 mercados políticos activos
- Para cada mercado:
  * Obtener precio actual de Polymarket CLOB API
  * Guardar price_snapshot en Supabase
  * Calcular cambio de precio vs último conocido
  * Si cambio > 5% → marcar mercado para re-análisis
    con force=True
  * Actualizar last_price y last_price_checked_at
- Procesar nuevas noticias RSS
- Si noticia nueva relevante → marcar mercados
  relacionados para re-análisis con force=True
- Aplicar filtro cuantitativo a mercados en cola
- Actualizar P&L unrealized de posiciones abiertas
- Ejecutar check_exit_rules()
- check_cooldowns() del key_manager

LOOP 2 — Análisis LLM (continuo, consume la cola):
- Cola de prioridad: primero mercados con force=True,
  luego mercados nuevos, luego caché expirada
- Para cada mercado en cola:
  * Ejecutar full_analysis(market, price, news, force)
  * Si should_trade → abrir paper trade
  * Guardar análisis en Supabase
  * Actualizar last_llm_analysis_at en el mercado

Log de observabilidad al final de cada scan de 5 min:

ENDPOINTS NUEVOS en server.py:
GET    /api/settings/keys
  → lista enmascarada de todas las keys (sin key_value)
POST   /api/settings/keys
  body: {service, key_value, label}
  → validar key haciendo llamada de prueba mínima
  → guardar en Supabase
  → llamar key_manager.reload_keys()
  → return {ok, id, label, service}
DELETE /api/settings/keys/:id
  → eliminar de Supabase y de memoria del key_manager
PATCH  /api/settings/keys/:id
  body: {is_enabled}
  → actualizar en Supabase y en memoria
GET    /api/settings/keys/status
  → estado actual de todas las keys en memoria:
    calls_today, in_cooldown, cooldown_until
